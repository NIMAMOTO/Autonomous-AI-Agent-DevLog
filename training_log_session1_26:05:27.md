## ② `def` の行を「輸入・輸出」で解釈

`def get_company_info(company_name: str) -> dict:` を自力で解釈した。

> 💬 「defというシグナルで、company_nameという輸入した値をstr（文字列）として捉え、dictで輸出すること。defが出たら、輸入した値を書式変えて輸出することで合っている？」

→ 方向性（入力・出力の発想）は正解だが、`def` の役割だけ惜しかった。`def` は「書式を変える命令」ではなく「関数を建設する宣言」。書式を変えるのは箱の中身のコード。

---

## ③ 「外枠と中身の分離」に自力で到達

修正を受けて、さらに踏み込んで言語化した。

> 💬 「大枠ってこと？defはただ箱自体と入り口・出口を作っただけ。なぜ文字列が辞書に転換できるかは知らない。」

→ これはソフトウェア工学でいう**「インターフェース（外枠）」と「実装（中身）」の分離**そのもの。初学者の段階で自力で切り分けたのは論理的に正しいアプローチ。

---

## ④ 辞書の単体練習でつまずく（print の罠）

`dict` を単体で練習。`print(profile)` を実行したら `<function profile at 0x...>` という暗号が出た。

→ 原因は2つ。①関数名に `()` を付けず名前だけ書いたため「機械そのもの（設計図）」が出力された。②`return` がなく、データが箱の中に閉じ込められていた。

→ 修正後、`return dummy_data` と `profile("tesuo")` で正しく辞書が出力された。

---

## ⑤ 引数を動的に組み込む

修正の際、`"name": aaa` として引数をそのまま辞書に組み込んだ。

→ 固定文字列ではなく、外部入力（"tesuo"）を変数として内部データに反映させる、プログラムのコア機能を正しく実装できていた。

## ⑥ 「自販機なのに固定値しか返さない」という違和感

Tool化に進む前に立ち止まった。

> 💬 「自販機というなら違う値を入れたら自動でタイプを分けてくれるはずだが、今は固定値しか返さない。異なる値を判断する基準を、クラウドLLMに任せるのか？」

→ この仮説は**エージェント設計の核心**。LLM（脳）が判断・条件分岐を担い、Tool（手足）は確定的な処理に専念する、という役割分担の発想に自力で到達した。

---

## ⑦ docstringエラーwith `@tool`

`@tool` を付けて実行したら `ValueError: Function must have a docstring` が出た。

→ `@tool` を付けた関数には、LLM向けの取扱説明書（docstring）が必須。これがないとLLMは「いつこのツールを使えばいいか」判断できない。`"""..."""` を1行足して解決。

→ Tool化した関数は、テスト実行も `profile("tesuo")` ではなく `profile.invoke({"aaa": "tesuo"})` を使う。

# PART 2｜知識・経験の分類

## 🔧 A. 技術知識（コード・文法）

### 関数（def）とは
自動販売機のイメージ。入れるもの（引数）→ 中身の処理 → 出てくるもの（戻り値）。

```
入れるもの（引数） → [ 自動販売機の中身（処理） ] → 出てくるもの（戻り値）
company_name      →  情報をdictにまとめる        →  完成したdict
```

```python
def get_company_info(company_name: str) -> dict:
```

| パーツ | 役割 |
|---|---|
| `def` | 「関数を建設する」という宣言（definitionの略） |
| `get_company_info` | 関数の名前（自由に決めてよい） |
| `(company_name: str)` | 入り口のルール：文字列を受け取る |
| `-> dict:` | 出口のルール：辞書を返す |
| `:` | 忘れるとエラー |

> ⚠️ **`def` の行は「外枠」を作るだけ。** なぜ文字列が辞書に変換されるかは、`def` の行自体は関知しない。変換するのは箱の中身のコード。これを「インターフェース（外枠）と実装（中身）の分離」と呼ぶ。

### 辞書（dict）とは
複数のデータをラベル付きでひとまとめにする形式。LangChainのエージェントが最も扱いやすいデータ形式。

```python
dummy_data = {
    "name": aaa,      # "キー": 値。引数を動的に組み込める
    "age": "23",
    "hobby": "ship"
}
```

- `{ }` で囲む。`"キー": 値` を `,` で繋ぐ。

### return とは
箱の中のデータを外の世界に届ける配達命令。これがないとデータは関数の中に閉じ込められたまま消滅する。

### インデント（字下げ）のルール
```python
def profile(aaa: str) -> dict:
    dummy_data = { ... }  # ← スペース4つ分、右にズレている
    return dummy_data     # ← これも同じズレ幅
```
- `def` の下の行は必ず**スペース4つ分**右にズラす
- Pythonは「このズレている部分が、関数のグループ」と判断する

> ⚠️ ズレが狂うと `IndentationError`。スペース4つ分に統一する。

### 関数の呼び出し
- 名前だけ（`print(profile)`）→ 機械そのもの（`<function...>`）が出る
- `()` を付ける（`profile("tesuo")`）→ 中の処理が動き、結果が返る

### @tool と docstring
普通のPython関数を、LLMが認識できる「手足（Tool）」に変えるデコレータ。

```python
from langchain_core.tools import tool

@tool
def profile(aaa: str) -> dict:
    """名前を入力すると、プロフィールを辞書形式で返すツールです。"""  # LLM向けの必須の取扱説明書
    ...
    return dummy_data
```

> ⚠️ `@tool` を付けたら docstring は**必須**。ないと `ValueError: Function must have a docstring`。  
> ⚠️ Tool化した関数のテスト実行は `profile.invoke({"aaa": "tesuo"})` を使う。

### 単語帳（語源で覚える）

| 単語 | 由来 | 意味 |
|---|---|---|
| `def` | definition | 定義する |
| `str` | string（糸・ひも） | 文字列 |
| `dict` | dictionary | 辞書 |
| `decorator` | デコレータ（室内装飾家） | 関数やクラスの前後に特定の処理を追加する機能 |
| `ValueError` | Value（値）の Error | 型は合っているが値の内容が不適切なときのエラー |
| docstring | Documentation Strings | 関数・クラス・モジュールの直下に `"""` で囲んで書く説明文。コードの機能や使い方を伝え、可読性を上げる |

## 🏗️ B. アーキテクチャ・設計の知識

### Python Script × LangChain × LLM の3者の関係

| 役割 | 例え | 何をするか |
|---|---|---|
| Python Script | マザーボード・OS | 全体の土台。処理の流れを制御 |
| LLM | クラウド上の超高性能CPU | テキストを読んで「次に何をすべきか」を推論。クラウド上に独立して存在し、PCの中には入ってこない |
| LangChain | 翻訳機・配線 | ローカルのPythonとクラウドのLLMを繋ぐ |

> 💡 **LLMはPython Scriptの判断の一部。** LLMは「Pythonに組み込まれた、高度な条件分岐（if文）の代わり」として機能する。ルールベースで処理できない曖昧な入力が来た時だけ、LangChain経由でLLMに投げ、判断結果だけを持ち帰る。

### LLM（脳）と Tool（手足）の役割分担
- **LLM（脳）：** 文脈理解・条件分岐・判断
- **Tool（手足）：** 確定的・機械的な処理（DB検索、Web取得など）

Toolは「特定機能に特化したシンプルなAPI」として設計し、複雑な状況判断はLLMに任せるのがベストプラクティス。

### なぜLangChainを呼ぶとAIが動くのか
LangChainがなければ、PythonからLLMにツールを使わせるのに以下を手動で書く必要がある。

```
LangChainがない世界（手動でやること）：
1. ツールの説明書（docstring）をLLMが理解できるJSON形式に変換
2. APIサーバーにHTTPリクエストを飛ばす
3. LLMが返した文字列を解析して、対応する関数を探し出す
4. その関数を実行する

LangChainがある世界：
→ 上記をすべて全自動でやってくれる
```

### ReActループの所在
ReActループ（思考→行動→ローカル実行→観察→思考→完了）は、**LangChainの内部にPythonの`for`/`while`文として内蔵されている**。`create_agent` の裏側に隠されているだけ。

**ReActの流れ：**
```
思考（LLM）→ 行動（どのToolを使うか決定）→ ローカル実行（PythonがToolを動かす）
    ↑                                                         |
    └────────── 観察結果をhistoryに繋いでLLMに再送 ←──────────┘
                        ↓
                  「最終回答」が出たらループ終了
```

```python
# LangChain内部のイメージ（擬似コード）
def create_agent_内部(llm, tools, user_input):
    history = [user_input]            # 会話履歴を鎖のように繋ぐ
    for i in range(max_iterations):   # ← ループの主導権は100%Python側
        llm_response = llm.invoke(history)
        if "最終回答" in llm_response:
            return llm_response
        observation = execute_tool(...)
        history.append(observation)   # 結果を繋いで次のループへ
```

> 💡 **なぜ「Chain（鎖）」か：** LLMはステートレス（記憶を持てない）。だからLangChainが思考と観察結果を毎回テキストの鎖として繋ぎ直し、丸ごとLLMに送り直して記憶を維持させている。ループの主導権はAI側ではなくPython側にある。

> 💡 **ループが見えなかった理由：** 簡単な問題は2周で終わるため一瞬で完了する。複雑な条件分岐が必要な問題ほど、ループが何度も回る。

## 🧠 C. 学習法・メタ認知（経験知）

### 自分で設計した5ステップ学習フレームワーク
```
0. このコードは何を作るためのもの？ 私は何をしたい？
1. 目的の把握：全体として何ができるか（全機能の洗い出し）
2. 全体の把握：機能 → パート（コード）の対応づけ
3. 個々の部分の把握：コード → そのコードが実現する機能
4. 個々の部分の訓練：単体で練習し、文法を馴染ませる。ノートを作る
5. 全体の訓練：全体を自力で書けるようにする
```
→ 全体から細部へ分解する「トップダウン学習法」。完成コードを拒否し、自分の理解のペースに合わせて分解する姿勢。

### 立ち止まる問いの価値
このセッションでは、解説を受け流さずに3回立ち止まった。
1. 「def は書式を変えるのか？」→ 外枠と中身の分離に到達
2. 「自販機なのに固定値しか返さないのは？」→ LLMとToolの役割分担に到達
3. 「なぜループができる？どこにある？」→ ReActループの所在に到達

→ いずれも誰にも促されず自分で立てた問い。この問いが理解を一段深いところへ運んだ。

### 概念を必ず最終目標に紐付ける（最重要の習慣）
> 概念を一つ学ぶたびに自問する：「これは自分のAgent demoの、どの行に対応するのか？」  
> （每写完一个概念，问自己一句："这个东西在我的Agent demo里，对应的是哪一行？"）

→ 学んだ知識を必ず最終目標（就活情報収集Agent）に紐付ける習慣。これが学習を「作業」で終わらせず、「目的に向かう手段」として位置づけ続ける核心。

### 詰まった時の対処法
```
理解できない行に当たったら：
→ 「このコードを日本語で説明して」とLLMに聞く
→ 説明を聞いたら「この部分を消したらどうなる？」と
   自分で予測してから実行する
```
→ 「予測してから実行する」が肝。これは後の「わざと壊す（ステップ4）」に発展する思考の原型。
