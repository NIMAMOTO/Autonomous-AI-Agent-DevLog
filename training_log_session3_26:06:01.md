# PART 2｜知識・経験の分類

## 🔧 A. 技術知識（ライブラリ・コード）

### requests とは
指定URLのサーバーに「手紙を出して、最初の一通目の返事（HTML）だけをもらって帰ってくる」高速なお使いロボット。

- ファイルを選んでいるわけではない。`requests.get('URL')` を実行すると、サーバーが決めた基本HTML（Documentファイル）を1つ受け取るだけ。
- ブラウザと違い、画像やJSファイルを追加で取りに行かない。**JavaScriptを実行できない**のが最大の限界。

```python
import requests
response = requests.get(url)   # responseは「通信結果が詰まった段ボール箱」
response.text                  # 箱の中身（HTML文字列）を取り出すカッター
```

### BeautifulSoup とは
`requests` が取ってきたHTML（長い1行の文字列）を、タグ構造のツリーに整理する「整理係」。

```python
from bs4 import BeautifulSoup
soup = BeautifulSoup(response.text, "html.parser")  # response.text（中身）を渡す
soup.get_text()        # タグを全部消してテキストだけ抽出
soup.find("script", type="application/ld+json")  # 特定タグを1つ狙撃
見つけたタグ.string     # タグの中身の文字列を取り出す（get_textだと消える場面で使う）
```

> ⚠️ **重要な落とし穴：** `get_text()` は「人間が読む文章だけ」を抽出する仕様のため、`<script>` や `<style>` タグの中身（JSON-LDなど）を**自動で捨てる**。

### 名前の不一致トラップ（bs4）
1つのライブラリで3つの名前を持つ。

| 場面 | 名前 |
|---|---|
| インストール（pip） | `beautifulsoup4` |
| import | `bs4` |
| コード内のクラス名 | `BeautifulSoup` |

```bash
pip install requests beautifulsoup4
```

### JSON-LD とは
`<script type="application/ld+json">` の形でHTMLに埋め込まれた構造化データ。**人間ではなく検索エンジン（SEO）のため**に企業が用意している。

- `requests` で取得可能（最初のHTMLに含まれるため）。
- キーと値が整理されているのでプログラムにとって最も扱いやすく、計算負荷も最小・最速。
- カバー率は採用ページで30〜40%程度。根本解決にはならない。

```python
import json
script_tag = soup.find("script", type="application/ld+json")
data = json.loads(script_tag.string)   # 文字列 → 辞書に変換
data.get("title", "なかった時の文字")    # 安全な取り出し
```

### パラメータ（引数）とは
関数（命令）に渡す「詳細設定」。命令が同じでもパラメータで結果が変わる。

```python
requests.get(url, headers=..., timeout=...)
# get=関数 / url, headers, timeout=パラメータ（注文の細かさ）
```

### 単語帳

| 単語 | 意味 |
|---|---|
| 引数（argument） | 関数に外部から渡すデータ（入力値） |
| HTTP POST | クライアントからサーバーへデータを送信するHTTPリクエストメソッド |
| query | 問い合わせ・質問 |
| encode | データ（文字・画像・動画）を扱いやすい別の形式に変換する作業 |
| parameter（param） | 引数・パラメータ |
| characters | 文字データ |

## 🏗️ B. アーキテクチャ・設計の知識

### JavaScriptの壁（動的生成）
現代のモダンなサイト（SPA）は、最初は空の骨組みだけを送り、JavaScriptが後からデータを描画する。`requests` はJSを動かせないため、描画前の空っぽの骨組みしか取れない。

### スクレイピング3つの突破ルート

| ルート | 方法 | カバー率 | コスト |
|---|---|---|---|
| A：静的解析 | JSON-LDや内部データを直接抽出 | 30〜40% | 最速・最軽量 |
| B：ヘッドレスブラウザ | Playwrightで描画を待ってDOM抽出 | 95%以上 | 重い・遅い |
| C：LLM特化API | Jina Reader / Firecrawl に外注 | ほぼ100% | 開発の手間ゼロ・外部依存 |

### フォールバック（段階的後退）アーキテクチャ
複数の取得ロジックを順番に試し、失敗したら次の手に自動で後退する設計。Googleのクローラー等で実際に使われる。

```
A（爆速・軽量）を試す → 失敗 →
B（最終兵器・重い）を起動
```
普段は0.1秒で高速に抜き、ガードの固いサイトだけ数秒かけて力技で突破する。速度と確実性の両立。

## 🧠 C. 学習法・メタ認知（経験知）

### 公式ドキュメント訓練法（自分で設計）

```
0. 知る（概念の定義）   ← 後から追加
   未知の専門用語の意味と背景を先に押さえる

1. 知る   AIに目的関数と公式ドキュメントのURLをもらう
2. 見つける   自力でドキュメントから答え（目的関数）を見つけ出す
3. 広げる   見つけた周辺を読む
   → 「この関数はなぜこのページにある？」
   → 「なぜこのページがドキュメントのこの位置にある？」
4. 壊す   理解したらわざと壊す
   「このコードが生きている理由を、死なせることで確認する」
   ・関数を消す → エラーを読む
   ・引数を変える → 何が変わるか見る
   ・順番を入れ替える → 動くか確認する
5. 作る   自分のコードに組み込む = 理解が本物になる瞬間
   理解する → 設計する → 証明する（GitHubに上げる）
```

### 方法A vs 方法B
- **方法A**（答えを先に知ってから確認）：迷わない・挫折しにくいが、見つける力は鍛えられない。
- **方法B**（全部自力で探す）：実務に最も近いが、何を探すかすら分からない段階では地図なしで歩かされる状態になる。
- → **今の段階は方法A。** ステップ0で「言葉の地図」だけもらい、歩くのは自力、という調整版が最適。

### 「知らない言葉は探せない」
問題文の言葉とドキュメントの言葉が繋がらないのは能力不足ではなく、経験の蓄積で解決するもの。知らない言葉に当たったら「答え」ではなく「言葉の意味だけ」をAIに聞く。

### 広がる視点 と 深く見る視点
両方とも本物の知性の動きだが、**ほとんどの場合は相反し、同時には使えない**。今どちらの場面かを判断する切り替えこそが技術。深く掘った経験は、次に広がる時の「解像度」になる。

## 🐛 D. デバッグで得た文法の教訓

| ミス | 教訓 |
|---|---|
| `chatollama` | クラス名は頭文字大文字（`ChatOllama`） |
| `from requests import get` なのに `requests.get()` | importの書き方と使い方は対応する。`import requests` なら `requests.get()` |
| `requests.get(url:'...')` | 引数で受け取った `url` に自分で値を書き込まない。AIが入れてくれる |
| `text` と `text_content` の混在 | 変数名は最後まで一致させる |
| `BeautifulSoup(response)` | 箱（response）ではなく中身（`response.text`）を渡す |
| `create_agent(...)` に名前なし | `app = create_agent(...)` で受け取らないと後で呼べない |

> 💬 エラーの種類が全部「コード全体の流れは見えているが、各行の細部が手に馴染んでいない」という同じ根から来ていた。構造は理解できているが、文法がまだ体に入っていない段階。
