## ⑩ 真因の確定：ハイドレーション

`"募集背景" in response.text` を実行。

> 結果：`募集背景が含まれています`（True）

→ 本文は初回HTMLに存在する。だが生HTMLを目視すると、「募集背景」は `<script type="application/ld+json">` の中（と、第三者AIが指摘した `data-react-props` 属性の中）にしか無い。
→ **真因：本文の文字列は (1) 属性値の中（get_textは属性を拾わない）と (2) scriptの中（get_textは捨てる）にしか無く、get_textが拾える普通のテキストノードとしての本文は初回HTMLに存在しない。**
→ この方式の正式名称が **ハイドレーション**（サーバーが初期データをHTMLに埋め込み、JS=Reactがそれを読んで画面を描く）。「データは静的HTMLにある／見た目はJSが描く」が同時成立する。

→ Session 3の「JSが描いている」という直感は**正しかった**。間違っていたのは「JSが描く＝requestsでは永遠に取れない」の部分だけ。今回はデータが同梱されているので取れた。
→ 戦略A（JSON-LD狙撃）は、get_textでは本文が一文字も取れないこのページに対する、唯一にして正しい解法だったと現物で裏付けられた。

## ⑫ 判定の境界線：確定的判定 vs 確率的判定（最重要の設計到達）

「各層の失敗をどう判定するか」を詰めた。ガイド役が「生HTMLと抽出結果を比較してPythonが門番せよ」と提案したが、**自分で盲点を指摘して却下した。**

> 💬 「最初にJSON-LDを叩く手順なら、get_textに来た時点でJSON-LD無しは確定済み。`in` での比較は不要。そして肉眼で確認できる範囲に限るなら、get_textが完全かをPythonが確定的に判定するのはほぼ不能だ。」

→ ガイド役の「Python門番」案は、JSON-LDのある特殊ケースでしか効かず、普通の静的サイトでは空振りすると判明。**この局面ではガイド役の方が間違っていた。**

到達した境界線：
- **JSON-LDの有無** → Pythonが確定的に判定できる（find_allが空か）。
- **get_text以降の完全性** → Pythonには判定不能。LLMに委ねるしかない。
- ただしLLM判定は「取れたものしか見られず、取れなかったものを知らない」という原理的限界がある（自分が①③で陥った罠と同じ）。

**LLM判定の危険を緩和する2つの自案：**
1. **強制要約 → 要約と目的を比較**：生データではなく、LLMが理解・要約した内容を、プロンプトの目的と照合。ずれを可視化する。緩和であって解決ではない。
2. **MCP/Agent Skillで「理想フォーマット」を持たせる**：就活情報の理想形（職名・業務内容・応募要件・給与・勤務地…）を基準として与え、「何が欠けているか」を欠けたもの基準で判定させる。今日自分が「53文字はおかしい」と気づけた暗黙の基準を、明示的にエージェントへ渡す発想。

→ 設計思想：LLM判定を完璧にしようとするのではなく、「**外れても致命傷にならない構造**」にする。誤判定の最悪ケースが「ベストではないデータを返す」で収まれば許容できる。

# PART 2｜知識・経験の分類

## 🔧 A. 技術知識（ライブラリ・コード）

### in 演算子（有無の確認）
`"探す文字列" in 対象の文字列` → True / False。ある文字列が別の文字列に含まれるかを一発で判定する基本演算子。今日、JS描画説の棄却・JSON-LD有無・本文の所在、3つの真実をこれ一つで確定させた。

### find と find_all
両方とも「タグを名指しで掴むピンセット」。get_text（全部の葉を拾う掃除機）とは正反対の道具。
- `find(...)` → 条件に合う最初の1個。
- `find_all(...)` → 条件に合う全部をリストで返す。引数の書き方は同じ。

> ⚠️ `findAll`（大文字）は旧名で DeprecationWarning が出る。正式名は `find_all`。
> ⚠️ `find_all` の結果はリスト。`BeautifulSoup(tags, ...)` のように箱ごと渡すと NoneType エラー。中身が欲しいなら `tags[0]` で取り出す。

### リストと辞書：取り出し方が違う

| | リスト（棚） | 辞書（ラベル付きの箱） |
|---|---|---|
| 姿 | `[枝0, 枝1]` | `{"title": "...", ...}` |
| 取り出し方 | **番号** `tags[0]`（0始まり） | **ラベル** `data.get("title")` |

> ⚠️ 辞書を `for x in 辞書:` で回すと、値ではなく**キー（ラベル）**が返る。
> ⚠️ 辞書に `[0]` は使えない（番号の概念が無い）→ KeyError。

### json.loads（文字列 → dict）
`loads` = load string。JSON形式の文字列を、本物のPython辞書に変換する翻訳機。

```python
import json
json_data = json.loads(tags[0].string)   # 文字列 → dict
json_data.get("title")                    # dictになって初めて .get が効く
```

> ⚠️ get_text() は HTML のタグを剥がす道具。JSON は HTML ではないので json.loads が要る。扱う「言語」が違う。

### 道具の使い分け（今日の混同ポイント）

| 道具 | 何から何へ | いつ |
|---|---|---|
| `.string` / `.get_text()` | タグ → 中の文字列 | 枝から中身を取り出す |
| `json.loads()` | JSON文字列 → dict | その文字列がJSONのとき |
| `BeautifulSoup(...).get_text()` | HTML文字列 → テキスト | descriptionの中のHTMLタグを掃除する仕上げ |

### 戦略A 完成形（調査スクリプト版）

```python
import requests, json
from bs4 import BeautifulSoup

def read(url: str):
    response = requests.get(url, timeout=30)
    return response.text

soup = BeautifulSoup(read(url), "html.parser")
tags = soup.find_all("script", type="application/ld+json")
json_data = json.loads(tags[0].string)

title = json_data.get("title")
description = BeautifulSoup(json_data.get("description"), "html.parser")
b = description.get_text()
```

> 📌 これは print で確認する調査用。エージェントの手足にするなら print ではなく **return（dictにまとめて）** が必要（Session 1の教訓）。未着手。

## 🏗️ B. アーキテクチャ・設計の知識

### 表示層とソース層
- **表示層**：ブラウザが解釈・描画した後の、整形された見た目。普段人間が見ている世界。
- **ソース層**：ブラウザが解釈する前の、生のHTML（タグ付き文字列）。requestsが触る世界。

F12（DevTools）を開く＝表示層からソース層へ降りる行為。「画面で見えるか」と「requestsで取れるか」が一致しないのは、層が違うから。

### 静的 / JS描画 / ハイドレーションの3パターン

| パターン | データの所在 | requestsで取れるか | 攻略法 |
|---|---|---|---|
| 静的 | 初回HTMLの普通のテキストノード | 取れる | get_text |
| ハイドレーション | 初回HTMLの属性・script内に同梱、JSが描画 | **取れる**（狙えば） | JSON-LD / 属性を狙撃 |
| 本当のJS描画 | 初回HTMLに無い。JSが後からAPIで取得 | 取れない | ヘッドレスブラウザ / API直撃 |

> 💡 **ハイドレーション**：サーバーが初期データをHTMLに埋め込み、JSがそれを読んで画面を組み立てる、今どきのReactサイトの定番。「データは静的にある／見た目はJSが描く」が両立。**今回のSmartHR/Talentioがこれ。**

### get_text() の死角（今日の最大の発見）
get_text() は2種類のものを拾えない：
1. **属性値の中**（`data-react-props="..."` など）→ 属性は拾わない。
2. **script / style の中身** → 仕様で捨てる。

ハイドレーションサイトでは本文がこの2つの死角にしか無いことがある。だから get_text() の結果がスカスカになる。「本文が無い」のではなく「get_textが拾えない場所にある」。

### ヘッドレスブラウザの正体
ブラウザの工程：HTTP通信 → DOM構築 → JS実行 → **画面にピクセルを描く**。
最後の「ピクセル描画」（一番重い）だけを省いたのがヘッドレスブラウザ。データを取るだけなら描画は不要だが、JSの書き込み先であるDOMの構築までは要る。

### 三層フォールバック（⑪参照）と判定の境界線（⑫参照）
- 確定的に判定できるもの（JSON-LDの有無）→ Pythonの責任。
- 確率的にしか判定できないもの（get_text以降の完全性）→ LLMに委ね、「外れても壊れない構造」で受ける。
- この「確定的か／確率的か」が、LLM（脳）とPython（手足）の境界線の引き方そのもの。

## 🐛 D. デバッグで得た教訓

| 現象 | 原因 | 教訓 |
|---|---|---|
| `findAll` で DeprecationWarning | 旧名を使った | 正式名は `find_all` |
| `'NoneType' object has no attribute 'find_all'` | `read(url)` が return 無しで None / `BeautifulSoup` をクラスから直接呼んだ | soupを作ってから `soup.find_all`。returnを忘れない |
| `len` が意図せず 2 | `find_all` の結果を変数で受けず、`len(soup)` を数えていた | 結果は変数で受ける。受けないと消える |
| `find_all("application/ld+json")` で取れない | 第1引数はタグ名。type値を置く場所がずれた | `find_all("script", type="application/ld+json")` |

| 辞書を for で回したらキーしか出ない / `json_data[0]` でエラー | リストと辞書を混同 | リスト=番号、辞書=ラベル(.get) |
